# lab-nidaq-mcp 安全設計

この文書は実装に先立って確定させた安全設計である。実装はここから逸脱してはならない。逸脱が必要と判断した場合は、実装ではなくまずこの文書を改訂する。

対象は **ソフトウェアタイミングの単点入出力**のみ。波形取得・連続ストリーミング・外部トリガ・カウンタ/タイマは対象外（末尾「対象外」参照）。

## 1. 対象ハードウェアの実測事実

2026-07-20 に実機から取得（DAQmx 25.5.0、`nidaqmx` 経由）。

| | USB-6210 (`Dev1`) | USB-6009 (`Dev2`) |
| --- | --- | --- |
| AI | 16 ch | 8 ch |
| AI レンジ | ±0.2 / ±1 / ±5 / ±10 V | ±1 / ±1.25 / ±2 / ±2.5 / ±4 / ±5 / ±10 / ±20 V |
| AO | **0 ch** | **2 ch (`ao0`, `ao1`)** |
| AO レンジ | — | **`[0, 5] V`** |
| AO 最大レート | — | 0 S/s（ソフトウェアタイミングのみ） |
| DI | 4 (`port0/line0-3`) | 12 (`port0/line0-7`, `port1/line0-3`) |
| DO | 4 (`port1/line0-3`) | 12 (`port0/line0-7`, `port1/line0-3`) |
| DIO の性質 | **DI と DO が別ポート** | **DI と DO が同一ライン（双方向）** |

この表の最後の 2 行が本設計の中心である。

## 2. 想定するハザード

- **H1 — AO 出力**: アナログ電圧が機器を駆動する。
- **H2 — DO 出力**: デジタルラインが機器を駆動する。
- **H3 — インターロックの自己無効化**: USB-6009 では DI と DO が同一ラインなので、インターロック用に読んでいるラインを出力として宣言でき、インターロックが自分自身を無効化し得る。USB-6210 では物理的に起こらない。
- **H4 — ホスト消失**: ホストのクラッシュや USB 抜去の後も出力は最後の状態で残る。**これらの機体は電源投入時出力状態のプログラム設定に対応しない**（X Series / cDAQ の機能）。したがってホストが消えた後の出力レベルをソフトウェアで保証することはできない。
- **H5 — 「0 が安全」の誤前提**: USB-6009 の AO レンジは `[0, 5] V` で、**0 V はレンジの下端であって中立点ではない**。制御対象によっては 0 V が全開・全速・遮断解除を意味し得る。「安全のため 0 を出す」は一般には成立しない。

## 3. 設計判断

### D1. 出力はオプトイン（既定は読み取り専用）

設定で宣言されないチャネル・ラインは出力として存在しない。出力を 1 つも宣言しない構成では、backend は lab-ble-mcp と同じく読み取り専用として振る舞う。

### D2. 宣言された出力は安全値の宣言が必須（コードに既定値を持たせない）

H5 のため、安全値をコードが推測してはならない。AO チャネル・DO ラインを出力として宣言するとき、`safe_value` の宣言を必須とし、欠落は設定読み込み時に拒否する。**`0` を暗黙の既定値にしない。**

### D3. インターロックに指定したラインは出力として宣言できない

H3 への対処。インターロックとして宣言されたラインが出力宣言にも現れた場合、設定読み込み時に拒否する。USB-6009 でのみ現実に起こるが、機体によらず一律に禁止する。

### D4. 出力前にインターロックを確認し、読めなければ拒否する

インターロックが宣言されている場合、**すべての出力書き込みの直前**に該当ラインを読み、要求値と一致しなければ書き込みを拒否する。読み取り自体が失敗した場合も拒否する（fail-closed）。インターロックが壊れているときに出力が通ってはならない。

インターロックが配線されていない構成もあるが、その場合は設定で **`interlock: none` を明示的に宣言させる**。宣言の欠落（書き忘れ）と、意図的な不設置とを区別できないまま出力を許すことはしない。

### D5. 出力を宣言した機器定義は `safe_shutdown` が非空必須

lab-ble-mcp では読み取り専用のため `safe_shutdown: []` を明示したが、本 backend では逆に、出力を宣言した機器定義が空の `safe_shutdown` を持つことを禁止する（テストで強制）。

### D6. 値域はチャネルごとに宣言し、範囲外はハードウェアに届く前に拒否

AO の書き込み値は、宣言された範囲と機体の物理レンジ（USB-6009 は `[0, 5] V`）の**両方**を満たさなければならない。検証は DAQmx タスクを作る前に行う。

### D7. 書き込みは自動リトライしない

既存のエコシステム規約。読み取りのリトライと書き込みのリトライを分離し、書き込みは既定でリトライしない。二重書き込みは二重動作を意味し得る。

### D8. `close()` は安全値へ駆動してから閉じる

`close()` は宣言済みのすべての出力を安全値へ駆動し、その後タスクを閉じる。BEF 契約により同期・冪等・例外を投げない。安全値の駆動に失敗しても例外を送出せず、失敗した事実は記録する。

### D9. ハードウェアの電源投入時状態は保証しない（設置要件として文書化する）

H4 のため、ソフトウェアは「ホスト消失後の出力レベル」を約束しない。代わりに、**出力ラインが高インピーダンスになったときに安全側になるよう外部回路を配線すること**を設置要件として README と機器定義に明記する。これはソフトウェアの約束ではなく、設置側の責務である。

`support_level: verified` は、この配線要件を満たした実機で確認した場合にのみ宣言できる。

## 4. 設定の形

```yaml
backends:
  nidaq:
    devices:
      Dev2:
        interlock: none          # または {line: "port0/line7", require: 1}
        analog_inputs:
          ai0: {range: [-10, 10]}
        analog_outputs:           # 宣言しなければ出力は存在しない
          ao0:
            range: [0, 5]
            safe_value: 0.0       # 必須。既定値なし
        digital_outputs:
          port1/line0:
            safe_value: 0         # 必須。既定値なし
```

## 5. ワイヤ言語（案）

resource は**機器単位** `DAQ::<DeviceName>`（例 `DAQ::Dev2`）。チャネルはコマンド側で指定する。

| コマンド | 種別 | 意味 |
| --- | --- | --- |
| `READ AI <chan>` | query | アナログ入力を読む（V） |
| `READ DI <port>/<line>` | query | デジタル入力を読む（0/1） |
| `INFO <field>` | query | 機体情報 |
| `WRITE AO <chan> <volts>` | write | アナログ出力（宣言済みチャネルのみ） |
| `WRITE DO <port>/<line> <0\|1>` | write | デジタル出力（宣言済みラインのみ） |
| `SAFE` | write | 宣言済み出力をすべて安全値へ駆動する |

`SAFE` は `safe_shutdown` の実体であり、単独でも呼べる。

## 6. 対象外（別設計）

以下は現行の BEF 契約 `query() -> str` に乗らないため、本 backend の v0.1 では扱わない。

- 波形取得・連続ストリーミング（USB-6210 の 250 kS/s・16 ch）
- 外部トリガ、サンプルクロック同期
- カウンタ/タイマ

これらは BITalino（連続生体信号）と共通の「連続データが backend から出ていく経路が契約に無い」という課題であり、まとめて別途設計する。
